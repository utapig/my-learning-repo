## AI&app-development

## 基本方針
- 画面を綺麗に作成できるとモチベーションが上がるので、小さいアプリでも必ず画面まで作成する
- 通勤中もアウトプットを含めた学習をしたいので、クイズを出してもらいながら学習する
- コードは基本的にAIではなく自分で書き、理解することを重視する

## 学習内容

| 技術 | 理由 |
| :--- | :--- |
| java | 業務で使うから。また、プログラミング言語の基礎を学べるから。 |
| React | フロントに興味があるから。 |

## 学習方法

| 場面 | 内容 | ツール名 | 理由 |
| :--- | :--- | :--- | :--- |
| 通勤時間 | AIにメンターとなってもらい、クイズを出してもらいながら体系的に学習する。 | ChatGPT | 使い慣れているため |
| 休日 | ミニアプリをAIにサポートしてもらいながら開発する。UIデザインにはこだわる。 | github copilot | 多くのユーザーが使っているため & 業務でも使っているため |

### 通勤時間学習のプロンプト

```
あなたは私専属のプログラミングメンターです。

私の条件：
- SE2年目
- Javaの基礎知識あり
- フロントエンドに興味あり
- 通勤時間が長く、平日は電車で1時間学習可能
- AI時代を前提にした学習をしたい
- メソッド暗記中心ではなく、概念・設計・役割理解を重視したい
- ただし必要最低限の頻出APIは身体化したい
- 最終目標は「フロント + Javaバックエンド + AI活用」で個人開発できる状態

あなたの役割：
1. 学習カリキュラム作成
2. 毎日の課題出し
3. コードレビュー
4. 理解度チェック
5. 抽象化支援
6. 学習ログ管理

出力ルール：
- 常に「何を覚えるか」ではなく「何の問題を解決する技術か」を先に説明
- ライブラリ説明時は必ず以下形式

技術名：
解決する問題：
内部概念：
代表API：
よくある失敗：
AIへ委譲可能範囲：

- メソッド暗記を強制しない
- 頻出概念だけ反復する
- 課題は30〜60分単位
- 通勤学習用と実装学習用を分離
- 1週間ごとに振り返りを行う
- 理解不足なら前工程へ戻す
```

### React学習限定の方針

Javaでバックエンドを実装したTodoアプリにおいて、画面の実装をReactで行った方が学習効率がよさそうだと考えた。
そこで、Todoアプリの画面作成における学習計画を改めて立て、一旦この方針で学習することにした。

```
# 学習方針

■ 最終目標
React + Spring Boot + AIを活用し、実務レベルの個人開発ができるようになる。

■ 学習スタイル
- 1本のTodoアプリを24週間かけて実務レベルまで育てる
- 必要になった技術だけ学ぶ（教材を順番に進めない）
- 「何を覚えるか」ではなく「何の問題を解決する技術か」を理解する

■ 技術学習の順番
1. 技術名
2. 解決する問題
3. 内部概念
4. 代表API
5. よくある失敗
6. AIへ委譲可能範囲

■ 平日
【通勤（10〜30分）】
- 今日使う技術の概念理解
- 設計問題
- Reactコード読解
- API仕様読解
- スマホだけで完結

【実装（30〜60分）】
- その日の機能を1つだけ実装
- 小さく完成させる
- コードレビュー・設計レビューを受ける

■ 休日
- 機能をまとめて完成させる
- リファクタリング
- 振り返り

■ AIの使い方
1. 自分で設計
2. AIレビュー
3. 修正
4. 実装
5. AIレビュー
6. リファクタリング

※ コード丸投げは禁止。AIは設計・レビュー・壁打ち相手として活用する。

■ レビュー方針
- コードレビュー
- 設計レビュー
- State配置レビュー
- 命名レビュー
- 実務ならどう設計するかを解説

■ 学習ログ
毎回更新する
- 理解済み
- 実装済み
- 理解不足
- 次回課題

■ 学習ルール
- 理解不足なら前工程へ戻る
- 「動いた」ではなく「なぜ動いたか」を説明する
- 暗記より概念・設計・責務を重視する
- 頻出APIだけは実装を通して自然に身につける
```
